URGENT BUGFIX: 8->10 upgrade quebrava QSV + main12/main-422-* nao
existem + /assets continuava sendo escaneado + skip de baixo bitrate

  [BUGFIX CRITICO #1] Forced 8->10 upgrade quebrava QSV main10
  ------------------------------------------------------------
  ANTES (v6.2.0): _resolve_output_format SEMPRE elevava saida pra
         10-bit (out_depth = 10 mesmo pra source 8-bit). Resultado
         em QSV: input nv12 (8-bit) saia do decoder, encoder pedia
         main10 (10-bit) — format mismatch. ffmpeg cuspia:
           [hevc_qsv] Current profile is unsupported
           [hevc_qsv] Low power mode is unsupported
         e o worker caia pra libx265 (CPU) DESNECESSARIAMENTE.

  AGORA: out_depth match input. 8-bit fonte vira main + yuv420p.
         10-bit fonte vira main10 + yuv420p10le. O upgrade 8->10
         era teoria boa pro libx265 mas nao se traduz em QSV/NVENC.

  [BUGFIX CRITICO #2] main12 e main-422-* nao existem em hevc_qsv
  ----------------------------------------------------------------
  ANTES: 12-bit ou 4:2:2 output mandava `-profile:v main12` /
         `main-422-10` pro hevc_qsv. ffmpeg:
           [hevc_qsv] Unable to parse "profile" option value "main12"
         O fallback automatico tentava libx265 mas perdia ~2 ffmpeg
         launches falhos antes (~30s + audio retry + cpu retry).

  AGORA: worker detecta upfront via probe e forca encoder=libx265
         em qualquer um destes:
           - source chroma e 4:2:2 ou 4:4:4
           - source bit_depth > 10 (alvo seria main12)
           - preserve_chroma_422 ON e source 4:2:2
         Aviso "Forcing libx265 (CPU): <razao>. Job rodara ~10x
         mais lento" no log antes do transcode.

  [BUGFIX] /assets nao estava sendo pulado
  -----------------------------------------
  ANTES: v6.2.1 adicionou */assets/* aos defaults de exclude_patterns
         no config.py. Mas o config.yaml do usuario ja tinha
         exclude_patterns proprio que sobrescreve o default (Pydantic
         nao mescla listas). Resultado: ProRes 4444 12-bit em
         /podcast/assets/drunk canvas/render intermed/ ainda era
         baixado, scaneado e tentado transcode (que falhava).

  AGORA: scanner._process_file chama _path_has_assets_segment(path)
         como PRIMEIRA gate, antes de qualquer outra checagem. Se
         qualquer segmento do path (case-sensitive 'assets'/'Assets'/
         'ASSETS') bater, skip imediato. Independente de
         exclude_patterns — usuario nao pode acidentalmente desligar
         essa protecao.

  [FEATURE] Skip de baixo bitrate (YouTube downloads etc)
  -------------------------------------------------------
  Novo state SKIPPED_LOW_BITRATE + setting
  low_bitrate_skip_mbps_per_megapixel (default 3.0). Quando o probe
  retorna bitrate por megapixel abaixo do threshold, marca o job
  como SKIPPED_LOW_BITRATE e libera staging — sem transcode, upload
  ou reorganize.

  Threshold escala com resolucao:
    - 1080p (~2 MP) skip < 6 Mbps  (catches YT 1080p ~5 Mbps)
    - 4K (~8 MP) skip < 25 Mbps    (catches YT 4K ~20 Mbps)
    - A7siii XAVC-S 4K ~140 Mbps -> 17 Mbps/MP, NAO pula
    - Camera ATEM 1080p 50 Mbps   -> 24 Mbps/MP, NAO pula

  Set 0 pra desligar.

QSV HYBRID DECODE: 4:2:2 input ja nao cai pra libx265 puro

  [PERF] H.264 High 4:2:2 (A7siii XAVC-S-I) agora encoda em QSV
  -------------------------------------------------------------
  ANTES: cliente A7siii grava H.264 High 4:2:2 10-bit. O decoder
         do QSV em chips Intel comuns NAO LE essa variante de
         input — so High (4:2:0). O ffmpeg falhava com -22 e o
         worker, depois de duas tentativas, caia pra libx265 puro
         (CPU). Resultado: ~0.38x realtime, 4K em quase 3h por hora
         de conteudo.

  AGORA: detectado upfront via probe.chroma. Quando input chroma
         e 4:2:2 ou 4:4:4 e a gente esta querendo usar QSV/NVENC:
           - dropa `-hwaccel qsv` / `-hwaccel cuda` (decode em SW)
           - aplica `-vf format=yuv420p10le` (chroma reduction em SW)
           - mantem `-c:v hevc_qsv` (encode em HW)
         Bottleneck vira o decode SW (~50-100 fps em 4K H.264 num
         CPU forte) em vez do encode SW. ~5-10x mais rapido que
         libx265 puro — 1.5-3x realtime em vez de 0.38x.

         Quando preserve_chroma_422 e ON E source e 4:2:2, o
         worker continua forcando libx265 (CPU), porque ai precisa
         de chroma 4:2:2 no OUTPUT, e o encoder QSV nao suporta
         Main 4:2:2 — esse caminho lento e inevitavel.

  [POLISH] /assets/ adicionado ao exclude_patterns default
  --------------------------------------------------------
  Pasta de project resources (LUTs, plates, fonts, graficos) nunca
  deve ser scaneada. Adicionado *​/assets/* e *​/Assets/* aos
  defaults de exclude_patterns no config.

MINOR BUMP: ATEM ._ sweep periódico + roteamento bit-depth/chroma + color
metadata passthrough

  Três entregas relacionadas, todas independentes mas cobrindo o
  pipeline antes da chegada das pastas A7siii (10-bit 4:2:2).

  [A] Sweep periódico de ._ + botão manual
  ----------------------------------------
  ANTES: cleanup_dot_underscore só rodava DENTRO do batch reorganize
         per-folder. Se um ._ chegava no Dropbox depois do batch (lag
         de sync), ele ficava orfão indefinidamente.

  AGORA:
    - Sweep periódico no scan loop. Cada N scans (default 1) o daemon
      varre dropbox_root recursivamente, identifica pastas matching
      'Video ISO Files' / 'Audio Source Files', e roda cleanup nelas.
      Config: cleanup_dot_underscore_sweep_every_n_scans (0 = desliga).
    - Endpoint POST /api/cleanup-dotunderscore-now + botão "Sweep ._
      now" no dashboard. One-shot manual pra resolver backlog.

  [B] Roteamento bit-depth / chroma com toggle Preserve Chroma 4:2:2
  ------------------------------------------------------------------
  ANTES: builder usava regra simples: 10-bit no input -> main10,
         senao main. Sempre yuv420p (4:2:0). Nao havia opcao pra
         preservar 4:2:2 do source.

  AGORA: novo metodo _resolve_output_format(probe) que decide:
    - Bit-depth nunca downgrade. 8 vira 10 (x265 comprime melhor em
      10-bit). 12 fica 12. >12 cap em 12 com warning bitdepth_capped_at_12.
    - Chroma: 4:2:0 default. Se preserve_chroma_422 ON E source 4:2:2,
      entao output 4:2:2. Source 4:4:4 trata como 4:2:2 com warning
      chroma_444_downgraded.

  Tabela de output (depth, chroma):
       (10, 420) -> yuv420p10le, profile main10
       (10, 422) -> yuv422p10le, profile main-422-10
       (12, 420) -> yuv420p12le, profile main12
       (12, 422) -> yuv422p12le, profile main-422-12

  Quando o output e 4:2:2, o worker DETECTA o source (probe) ANTES de
  chamar o builder e FORCA encoder=libx265 (CPU). QSV e NVENC consumer
  nao implementam HEVC Main 4:2:2, entao tentar QSV ali quebra.

  Toggle no dashboard:
    - Cor laranja chamativa, label começa com "⚠"
    - Hint expande explicando ~10x mais lento
    - Modal de confirmacao na primeira vez que liga ("vc tem certeza?
      Vai virar 5-10h um clip de 1h.")
    - Default OFF — workflow padrao 4:2:0 em todos os jobs.

  [C] Color metadata passthrough
  ------------------------------
  ANTES: ffmpeg_builder nao passava color flags. Confiava no
         propagacao automatica do ffmpeg, que funciona OK pra Rec.709
         mas falha em log/raw (S-Log3). Sony em S-Log3 nao escreve
         color_primaries / transfer / space — so color_range='pc'.
         Sem passthrough, o output ficava com tags inferidas erradas
         e o NLE renderizava errado.

  AGORA: prober extrai color_primaries, color_transfer, color_space,
         color_range e normaliza 'unknown'/'unspecified'/'reserved'
         para None. Builder passa cada flag SO se o input declarou:
           if probe.color_primaries: -color_primaries <X>
           if probe.color_transfer:  -color_trc <X>
           if probe.color_space:     -colorspace <X>
           always: -color_range <X or 'tv'>

         Resultado:
           - Rec.709 (bt709) -> 4 flags setados, output mantem bt709
           - S-Log3 (Sony A7siii) -> so -color_range pc, outros
             omitidos. Curva log preservada, NLE aplica LUT como
             sempre.
           - Zero conversao de cor: nao passamos zscale ou colormatrix.

         Heuristica info-log: se 10-bit + primaries/transfer/space
         todos None + range 'pc', loga "detected likely log/raw
         source" pra confirmar pro operador que entendemos.

  Tambem adicionei probe extra: chroma ('420' | '422' | '444')
  derivado do pix_fmt. Disponivel via VideoInfo.chroma pra todo o
  pipeline.

CLEANUP: ._ macOS resource forks (lixo de 4 KB que a ATEM espalha)

  [FEATURE] Quarantine + auto-delete dos arquivos ._
  --------------------------------------------------
  Apos cada batch reorganize per-folder bem-sucedido, o daemon agora
  varre o `._*` (resource forks Finder, ~4 KB cada) que ficam
  espalhados pela pasta. Eles sao movidos pra subpasta
  `<parent>/ponto tracinho/`, e essa subpasta tem deletion agendada
  apos N segundos.

  Fluxo:
    1. Reorganize batch termina (15/15 swapped, por exemplo)
    2. Cleanup lista <parent>/ nao-recursivo, filtra entries que
       comecam com `._`
    3. Cria <parent>/ponto tracinho/ (idempotent)
    4. Move cada `._<arquivo>` pra la
    5. Agenda delete da pasta apos delay configurado
       (cleanup_dot_underscore_delete_after_seconds, default 300s)

  Configuracoes novas:
    - cleanup_dot_underscore: true/false (default true)
    - cleanup_dot_underscore_delete_after_seconds: int (default 300)
    - Ambas expostas no dashboard

  Best-effort: falhas individuais (move falha pra um arquivo) sao
  logadas mas nao abortam o pipeline. Arquivos quarantinados sao
  recuperaveis via Dropbox version history sempre.

URGENT BUGFIX: loop infinito de transcode (rename falhando no Windows)

  [BUGFIX CRITICO] FFmpeg termina, rename falha, repete pra sempre
  ----------------------------------------------------------------
  ANTES: workers._transcode_job() promovia o output via
         cmd.temp_output_path.rename(output_path). No Windows,
         Path.rename() levanta WinError 183 ("Cannot create a file
         when that file already exists") se o destino ja existe.
         Cenario: transcode anterior renomeou com sucesso, mas
         validation ou upload falhou e o job foi reenfileirado.
         O output.mp4 ficou no staging. Na retry, ffmpeg gera novo
         output.tmp.mp4, tenta rename, falha — e fica em loop
         infinito reencodando o mesmo arquivo, queimando QSV. O
         downloader 0 e o transcoder 0 ficavam presos no mesmo job.

  AGORA: usa cmd.temp_output_path.replace(output_path). replace()
         e o "rename atomico que sobrescreve" multiplataforma —
         no Windows usa MoveFileEx com MOVEFILE_REPLACE_EXISTING.
         Se sobrar um output.mp4 velho, e sobrescrito sem erro.
         Validation roda em seguida — entao corrupcao real e
         pega normalmente.

SCAN: preflight HEVC probe (range-download) + h264 deletion audit log

  [NOVO] Preflight HEVC detection antes do download completo
  ----------------------------------------------------------
  ANTES: arquivos nativamente em H.265 (entregues prontos, ou
         transcodificados fora desse daemon) eram baixados INTEIROS,
         o probe local descobria que ja era HEVC, marcava SKIPPED_HEVC
         e deletava o staging. Desperdicio gigante de banda — um
         arquivo de 100GB pra descobrir que nao precisava transcodar.

  AGORA: antes do download completo, o downloader faz range-download
         dos primeiros 16 MB do arquivo (configuravel via
         `preflight_hevc_probe_mb`) usando files_get_temporary_link
         + HTTP Range, e roda ffprobe no chunk. Se detectar HEVC,
         marca SKIPPED_HEVC e nao baixa o resto.

         Fallback: se o probe do head for inconclusivo (moov atom
         no final, comum em exports do Premiere), tenta os ultimos
         16 MB. Se ambos falharem, baixa inteiro como antes.

         Custo pior caso por arquivo desconhecido: ~32 MB. Ganho
         por arquivo HEVC nativo: ~99.97% da banda economizada.

         Default: 16 MB. Setar 0 desabilita totalmente.

  [NOVO] Audit log de delecao do /h264
  ------------------------------------
  Quando legacy_reorganize_delete_h264_after_seconds > 0 dispara
  a delecao da pasta /h264, o daemon agora primeiro escreve um
  log em <parent>/h264 deletado.txt listando:
    - filename de cada arquivo deletado
    - tamanho legivel
    - data/hora da delecao
    - motivo (delay configurado)
    - lembrete de recovery via Dropbox history

  Cada batch de delecao APPENDA uma nova secao no mesmo arquivo,
  entao o usuario tem o historico completo de tudo que ja foi
  apagado da pasta. Sobrevive ao rmtree do /h264 (esta no parent).

--------------------------------------------------------------------------------

UPLOAD: BUGFIX CRITICO - malformed_path no Dropbox

  [BUGFIX] Uploads de H.265 falhavam 100% com WriteError('malformed_path')
  -----------------------------------------------------------------------
  ANTES: workers._upload_job() construia o caminho da pasta de destino
         no Dropbox usando str(Path(job.output_path).parent). No Windows,
         pathlib.Path retorna WindowsPath e str() converte os '/' em '\'.
         Resultado: o cliente chamava files_create_folder_v2 com
         '\HeavyDrops\...\h265' e o Dropbox API rejeitava com
         malformed_path. O upload entrava em loop de retry (5 tentativas
         por uploader, 2 uploaders) e nunca subia o arquivo. Cada job
         transcodado virava lixo: GPU queimada, espaco em disco preso,
         saving zero.

  AGORA: usa PurePosixPath em vez de Path para construir o caminho de
         destino. PurePosixPath nao depende do OS e sempre retorna '/'.
         Defesa em profundidade: DropboxClient._normalize_path tambem
         converte qualquer '\' que escape para '/' antes de chamar a
         SDK, entao callers futuros nao reproduzem o mesmo bug.

    hd reorganize-existing --dry-run   # so mostra o plano, nao mexe em nada
    hd reorganize-existing --min-age-days 90

  [NOVO] Filtro por atividade da pasta (config: legacy_reorganize_min_age_days)
  ---------------------------------------------------------------------------
  Default: 60 dias. Pastas onde QUALQUER arquivo foi modificado nos ultimos
  N dias sao consideradas "ativas" e o swap NAO acontece, nem na conversao
  ao vivo nem no comando retroativo. Resolve o problema de mexer em pastas
  de projetos que voce ainda esta editando.

  Esse filtro vale pra:
    1. Conversoes ao vivo no daemon (se a pasta tiver atividade, o H.265
       fica em /h265/ ate a pasta ficar quieta).
    2. hd reorganize-existing (pula pastas ativas no plano e na execucao).

  Pra desligar o filtro completamente: legacy_reorganize_min_age_days: 0
  Pra reorganizar apenas pastas bem antigas: legacy_reorganize_min_age_days: 180

  [REFACTOR] Modulo src/transcoder/reorganize.py
  -----------------------------------------------
  Toda a logica de swap atomico (move original -> h264/, swap h265 ->
  spot original, append em feito.txt, rollback em caso de falha) agora

NOVA ARQUITETURA - "TRANSCODER 6"

Esta versão é uma reorganização completa pensada para arquivos massivos
(centenas de TB) rodando sem intervenção por meses. O app antigo
(transcoder_gui.py) continua funcionando para uso pontual em pastas locais,
mas o produto principal agora é um daemon de fundo com dashboard no navegador.

  [INSTALAÇÃO] Um comando, sem admin
  ----------------------------------
  Cole no PowerShell:
      iwr https://raw.githubusercontent.com/luisimperator/rep01/main/bootstrap.ps1 -UseBasicParsing | iex

  Instala em %USERPROFILE%\HeavyDrops, baixa Python/Git/FFmpeg se faltarem,
  pede o token do Dropbox uma vez, registra Task Scheduler para auto-iniciar
  no logon e reiniciar em caso de falha. Atalho "HeavyDrops" no Desktop abre
  o dashboard. Não precisa de admin.

  [DASHBOARD] http://127.0.0.1:9123/
  ----------------------------------
  Página simples no navegador, atualiza a cada 3s:
    - Versão instalada e tempo de uptime
    - Modo de scan (BULK na 1ª passada, DELTA depois)
    - Filas de download / transcode / upload
    - Disco reservado vs. livre
    - Lista dos 20 jobs mais recentes
    - Botões: Scan now, Pause, Resume, Retry FAILED
    - Banner se houver release nova no GitHub

  Fechar o navegador NÃO para o daemon.

  [SCANNER] Bulk + Delta com cursor
  ---------------------------------
  ANTES: cada scan re-listava todo o Dropbox.
  AGORA: 1ª passada (BULK) salva um cursor a cada 500 entradas e resume após
         crash. Passadas seguintes (DELTA) só trazem mudanças desde o cursor.
         Reinicia automaticamente se você mudar dropbox_root no config.

  [SCANNER] Stability dual-profile
  --------------------------------
  ANTES: todo arquivo esperava 45min de "min_age" antes de processar, mesmo
         em arquivo antigo que está estático faz meses.
  AGORA: BULK usa profile {checks=1, min_age=0} (o arquivo já está estático).
         DELTA usa o profile antigo {checks=3, min_age=900} para proteger
         de uploads em andamento. Reduz drasticamente o tempo da 1ª passada.

  [DISPATCHER] Filas centralizadas (fim do double-pickup)
  -------------------------------------------------------
  ANTES: cada worker fazia polling no SQLite com LIMIT 1 e disputava jobs.
  AGORA: uma thread só lê o DB e enche 3 filas bounded (download / transcode /
         upload). Workers consomem das filas com active_set central. Fim de
         duplicação em escala. Pause/resume sem matar jobs em andamento.

  [DISCO] Budget de staging (opt-in)
  ----------------------------------
  Para um disco local de 2TB apontando para 200TB de Dropbox, agora tem teto.
  O downloader pede slot antes de baixar; se não tem espaço (max_staging_bytes
  ou min_free_bytes), espera com log a cada 5min em vez de quebrar com ENOSPC.
  Reservas órfãs (crash com staging cheio) são limpas no startup.

  [LAYOUT] Mirror opcional
  ------------------------
  ANTES: saída sempre em /A/B/h265/clip.MP4 ao lado do original.
  AGORA: layout: sibling (legado, padrão) ou layout: mirror que coloca tudo
         em /_h265_output/A/B/clip.MP4 sob uma raiz única. Útil quando
         múltiplas pastas-irmãs têm arquivos com mesmo nome.

  [API] Rate limit do Dropbox
  ---------------------------
  Token bucket configurável (default 600/min, burst 50). O cliente espera em
  vez de levar 429. Em 429 honra o backoff retornado pelo Dropbox.

  [UPDATE] Notify-only com 1 comando
  ----------------------------------
  Daemon checa GitHub Releases no startup. Se tem versão nova, mostra banner
  no dashboard. "hd update" faz git pull + pip install -e . Aplicar continua
  sendo decisão sua.

  [DB] Índice unique parcial
  --------------------------
  jobs(dropbox_path) único restrito a estados não-terminais. Bloqueia jobs
  ativos duplicados para o mesmo path entre rev bumps, sem afetar histórico.

  [HOTFIX] UTF-8 forçado em Windows
  ---------------------------------
  Reads/writes de arquivo com encoding="utf-8" e errors="replace" para
  sobreviver a bytes não-ASCII em nomes de arquivo / paths do Dropbox.

COMPATIBILIDADE:
  - O GUI legado (transcoder_gui.py) continua incluído no zip e instalável
    via installer/install.ps1 para uso pontual em pastas locais.
  - O instalador novo (bootstrap.ps1) instala o daemon, não o GUI legado.

  hd-daemon   - mesmo daemon, nome usado pelo Task Scheduler
  hd-gui      - abre o dashboard no navegador padrão
